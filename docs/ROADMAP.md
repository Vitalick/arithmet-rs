# План развития

Этот документ фиксирует ближайшие доработки после базовой реализации:
адаптивный TUI, TOML-настройки, генерация упражнений, проверка ответов,
оценка и сохранение результатов уже есть в проекте.

## 1. Управление мышью

Добавить поддержку мыши в терминальном интерфейсе через `crossterm`.

- переключать арифметические действия кликом по списку действий;
- открывать редактирование имени, границ результата, количества примеров и
  времени на ответ кликом по соответствующей строке;
- запускать сеанс и переходить к следующему примеру кликом по центральной
  области или отдельной кнопке в статусной строке;
- прокручивать таблицу результатов колесом мыши;
- оставить клавиатуру основным и полностью рабочим способом управления.

## 2. Просмотр сохраненных результатов

Сейчас экран результатов показывает последний завершенный сеанс из памяти.
Нужно добавить просмотр результатов, сохраненных в файловой системе.

- читать каталог `results_dir` из `arithmet.toml`;
- находить результаты по пользователям и датам в подкаталогах `results/<user>/`;
- показывать общий список всех сохраненных запусков;
- добавить режим выбора по пользователям: сначала список имен, затем список
  результатов выбранного игрока;
- открывать выбранный JSON-результат в текущем виджете результатов;
- из открытого результата уметь применить настройки сеанса и повторить заход с
  теми же параметрами;
- обрабатывать отсутствующие, поврежденные или устаревшие файлы без падения
  приложения;
- сохранить текстовые `.txt` протоколы как человекочитаемый экспорт, а JSON
  использовать как основной формат для повторного просмотра в TUI.

## 3. Группировка результатов по настройкам

Добавить просмотр результатов, сгруппированных по одинаковым настройкам
тренировки без учета имени игрока. Это позволит сравнивать, как разные люди
проходили одинаковые наборы примеров и лимитов.

- построить ключ настроек без `player_name`: действия, границы результата,
  количество примеров, лимиты времени и будущие настройки новых операций;
- показывать список групп с кратким описанием настроек и количеством запусков;
- внутри группы показывать результаты всех игроков, отсортированные по дате или
  оценке;
- из группы уметь применить эти настройки для нового захода;
- явно показывать имя игрока в строках результатов, потому что оно исключено из
  ключа группировки;
- при расширении формата настроек обновлять ключ группировки так, чтобы старые
  и новые результаты сравнивались предсказуемо.

## 4. Переименование сложности

Текущий параметр `Сложность` фактически означает лимит времени на один ответ.
Нужно переименовать его в интерфейсе, README и протоколах.

- в UI показывать `Время на ответ`;
- в текстовом протоколе писать `время на ответ`, а не `сложность`;
- в коде переименовать `ActiveField::Complexity` в поле с явным смыслом;
- в TOML оставить `answer_time`, потому что это уже корректное имя.

## 5. Разное время для разных действий

Сейчас один `answer_time` применяется ко всем операциям. Нужно дать возможность
настраивать отдельный лимит для каждого действия.

- сохранить общий лимит как значение по умолчанию;
- добавить опциональные лимиты для `+`, `-`, `*`, `/`, `:`, а после появления
  новых действий также для `^`, корня и `%`;
- на экране упражнения брать лимит из операции текущего примера;
- в результатах показывать фактический лимит и затраченное время;
- продумать миграцию старого `arithmet.toml`: если отдельных лимитов нет,
  использовать общий `answer_time`.

## 6. Новые действия

Расширить список упражнений новыми типами, сохранив общий подход:
настройки задают диапазон результата, генератор создает корректные примеры,
а результаты сохраняются в JSON и текстовом протоколе.

### Степень `^`

- добавить операцию возведения в степень;
- в настройках указать, какие степени разрешены, например `2`, `3`, `4`;
- генерировать основание так, чтобы результат попадал в заданные пределы;
- исключить слишком большие значения, которые делают пример нечитаемым.

### Корень

- добавить операцию извлечения корня;
- в настройках указать разрешенные степени корня, например квадратный и
  кубический корень;
- генерировать только целочисленные ответы;
- показывать пример в понятном текстовом виде для TUI, например `sqrt(144)` или
  `root3(27)`, если символы корня плохо читаются в терминале.

### Остаток от деления `%`

- добавить отдельную операцию на остаток от деления;
- показывать ее как `%`;
- ожидаемым ответом считать остаток;
- подбирать примеры так, чтобы тренировать и деление, и вычитание: пользователь
  должен мысленно найти ближайшее кратное и остаток;
- не смешивать `%` с текущим `:`: `:` остается делением с остатком, где ответом
  является частное, а остаток показывается в правильном ответе.

## 7. Проверки

Для каждой доработки нужны точечные тесты.

- тесты сериализации и миграции настроек;
- тесты генераторов новых операций;
- тесты таймера для общего и per-operation лимитов;
- тесты загрузки сохраненных JSON-результатов;
- тесты группировки результатов по настройкам без учета имени;
- тесты применения настроек из выбранного результата или группы;
- ручная проверка TUI для клавиатуры и мыши.
