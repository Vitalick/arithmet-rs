# План переноса `arithmet_c` на Rust

Документ описывает, как переносить C-проект
`/home/vitaly/CLionProjects/ARITHMET` в текущий Rust-проект. Цель - прямой
перенос функционала с более переносимой архитектурой и адаптивным TUI.

Исходный репозиторий C-версии:
`https://github.com/Vitalick/arithmet_c.git`.

## 1. Исходное поведение C-версии

По текущему C-проекту функциональность разложена так:

- `main.c` - запуск программы, чтение и сохранение `a.dat`, главный экран,
  обработка горячих клавиш настроек, старт сеанса и просмотр результата.
- `app_state.c/.h` - глобальные настройки, имя игрока, баннеры, текстовые
  ресурсы, состояние результата сеанса.
- `session.c/.h` - генерация примеров, таймер ответа, проверка ответа,
  расчет оценки, заполнение результата.
- `input.c/.h` - ввод имени, числовых настроек и ответа; поддержка русского
  UTF-8 и перемещения курсора.
- `screens.c/.h` - главный экран, экран упражнения и экран просмотра результата.
- `protocol.c/.h` - сохранение текстового протокола, транслитерация имени в
  DOS-совместимое имя файла и выбор расширения `000`-`999`.
- `conio*`, `terminal_*`, `keyboard_mapping.c`, `keys.h` - DOS/conio-совместимый
  терминальный слой.
- `dos_compat*`, `asm*`, `mouse*` - совместимость с Turbo C/DOS и заглушки
  старого поведения.

В Rust переносить нужно не старый `conio` слой как архитектуру, а поведение,
которое он обеспечивает: экраны, цвета, горячие клавиши, таймер, ввод и
результаты.

## 2. Целевая архитектура Rust-проекта

Рекомендуемая структура:

```text
src/
  main.rs
  app.rs
  config.rs
  domain/
    mod.rs
    operation.rs
    settings.rs
    exercise.rs
    session.rs
    grade.rs
  protocol/
    mod.rs
    legacy.rs
    writer.rs
    filename.rs
  tui/
    mod.rs
    app_view.rs
    event.rs
    input.rs
    results_view.rs
    theme.rs
```

Назначение модулей:

- `domain` - чистая бизнес-логика без терминала и файловой системы.
- `config` - чтение и запись настроек в человекочитаемый файл.
- `protocol` - запись новых протоколов и чтение старых результатов.
- `tui` - терминальный интерфейс, адаптивная раскладка и обработка событий.
- `app` - состояние приложения и координация между `domain`, `protocol`,
  `config` и `tui`.

Такой разрез позволит тестировать генерацию примеров, оценку, настройки и
формат протокола без запуска терминального интерфейса.

## 3. Зависимости

Минимально полезный набор:

- `ratatui` - адаптивный TUI поверх терминала;
- `crossterm` - кроссплатформенный ввод, alternate screen, raw mode, resize,
  цвета и опциональная мышь;
- `serde` - сериализация настроек;
- `toml` - TOML-конфиг;
- `rand` - генерация случайных примеров;
- `chrono` или `time` - дата и время в протоколах;
- `anyhow` или `thiserror` - ошибки приложения.

Предпочтительный вариант для интерфейса: `ratatui + crossterm`. Он даст меньше
ручной платформенной работы, чем самостоятельный перенос `conio`.

## 4. Настройки

C-версия хранит настройки в бинарном `a.dat`:

- `0..4` - включенные операции;
- `5` - нижний предел результата;
- `6` - верхний предел результата;
- `7` - количество примеров;
- `8` - текущий ответ или служебный код;
- `9` - лимит времени в секундах;
- отдельно хранится имя пользователя.

В Rust это нужно заменить на TOML-конфиг, например:

```toml
player_name = "неизвестно"
results_dir = "results"

[operations]
addition = true
subtraction = false
multiplication = false
division = false
division_with_remainder = true

[limits]
result_min = 100
result_max = 150
exercise_count = 20
answer_time_seconds = 30
```

Правила валидации сохранить:

- хотя бы одна операция включена;
- `result_max - result_min >= 50`;
- количество примеров: `5..=200`;
- сложность: разумный диапазон, например `1..=99`, при прямом переносе можно
  сохранить `0..=99` и дефолт `30`;
- если имя пустое, использовать `неизвестно`.

Новый конфиг лучше хранить в каталоге программы или в стандартном
пользовательском config-dir. Для прямого и понятного поведения на старте
проекта проще начать с `arithmet.toml` рядом с исполняемым файлом или текущим
рабочим каталогом.

## 5. Модель предметной области

Рекомендуемые типы:

```rust
enum Operation {
    Addition,
    Subtraction,
    Multiplication,
    Division,
    DivisionWithRemainder,
}

struct Settings {
    player_name: String,
    operations: Vec<Operation>,
    result_min: i32,
    result_max: i32,
    exercise_count: usize,
    answer_time_seconds: u64,
    results_dir: PathBuf,
}

struct Exercise {
    left: i32,
    right: i32,
    operation: Operation,
    expected: i32,
}

struct AnswerRecord {
    exercise: Exercise,
    entered: AnswerInput,
    elapsed_seconds: u64,
    is_correct: bool,
}

enum AnswerInput {
    Value(i32),
    Escaped,
    TimedOut,
    SessionAborted,
}

struct SessionResult {
    settings_snapshot: SettingsSnapshot,
    records: Vec<AnswerRecord>,
    grade: u8,
}
```

Важно не тащить глобальный `app_settings[10]` в Rust. Лучше явно назвать поля:
так легче читать, валидировать и тестировать.

## 6. Генерация примеров

Алгоритм нужно перенести максимально близко к `session.c`.

Текущие правила:

- сложение: сначала выбирается результат в заданных границах, затем два
  ненулевых слагаемых;
- вычитание: выбирается уменьшаемое в границах, затем вычитаемое и результат;
- умножение: выбирается результат и подбираются множители без единиц;
- деление: выбирается делимое и подбирается делитель так, чтобы деление было
  целым;
- деление с остатком: сначала строится пример для деления, затем к делимому
  добавляется остаток.

На первом этапе лучше написать перенос буквально, но без `goto`: использовать
циклы `loop { ... break exercise; }` и отдельные функции на каждую операцию.

Обязательные тесты:

- результат каждого сгенерированного примера попадает в ожидаемые правила;
- для деления без остатка `left / right == expected`;
- для деления с остатком ожидаемый ответ совпадает с целой частью, а остаток
  находится в корректном диапазоне;
- генератор не создает нули и единицы там, где C-версия их отбрасывает;
- при включенных нескольких операциях выбор операции покрывает только
  разрешенные операции.

## 7. Оценка

Логику оценки сохранить:

- `>= 98%` - 5;
- `>= 75%` - 4;
- `>= 50%` - 3;
- `>= 20%` - 2;
- иначе 1.

Нужно отдельно тестировать:

- границы процентов;
- расчет текущей оценки после каждого ответа;
- итоговую оценку по плановому количеству примеров, как в C-версии.

## 8. Терминальный интерфейс

Главный экран должен быть похож по управлению, но не обязан повторять
координаты DOS-экрана.

Рекомендуемая раскладка:

- верхняя строка: название, версия;
- левая колонка: операции и их состояние;
- правая или верхняя область: имя, границы, количество, сложность;
- центральная область: текущий пример, поле ответа, верный ответ, проверка;
- нижняя область: подсказки горячих клавиш;
- экран результатов: сводка сверху, список ответов ниже, прокрутка по высоте
  терминала.

Адаптивность:

- при ширине меньше условного минимума показывать компактную версию;
- высоту списка результатов считать от фактической высоты терминала;
- не фиксировать 11 строк результатов, как в C-версии;
- на resize перерисовывать весь экран через `ratatui`.

Управление сохранить:

- `+`, `-`, `*`, `/`, `:` - включить/выключить операции;
- `И` или латинская fallback-клавиша `B` - имя;
- `О` или `J` - нижняя граница;
- `Д` или `L` - верхняя граница;
- `К` или `R` - количество примеров;
- `С` или `C` - сложность;
- `Enter` - старт/подтверждение;
- `Esc` - выход или отказ от текущего примера;
- `F10` - выход из текущего сеанса;
- `F1` - результаты;
- `Tab` - следующий пример после ответа.

Мышь:

- оставить как необязательное улучшение;
- если включать, использовать события `crossterm`;
- сначала поддержать клики по операциям и вкладкам/кнопкам, без сложной
  эмуляции DOS-мыши.

## 9. Результаты и протоколы

C-версия сохраняет протоколы в текстовые файлы с именем вида `NAME____.000`.
Имя формируется транслитерацией русского имени и заполнением до DOS-формата
`8.3`, расширение выбирается от `000` до `999`.

В Rust нужно поддержать два режима:

- запись нового протокола в понятном текстовом формате, максимально похожем на
  C-версию;
- чтение старых протоколов из каталога результатов и показ их в TUI.

Практичный план:

1. Сначала сохранить генерацию старого имени файла и старый текстовый формат.
2. Добавить настройку `results_dir`.
3. Реализовать список протоколов из `results_dir`.
4. Для старых файлов сделать tolerant parser: если файл не парсится полностью,
   показывать его как plain text.
5. После стабилизации можно добавить новый структурированный формат, например
   `json` рядом с текстовым протоколом, но текстовый протокол оставить для
   совместимости.

## 10. Платформенная независимость

Что не переносить буквально:

- `conio`;
- BIOS/DOS-прерывания;
- ручную эмуляцию видеостраниц;
- бинарный `a.dat`;
- платформенные `terminal_input_*` и `terminal_screen_*` как отдельную
  архитектуру.

Что использовать вместо этого:

- `crossterm` для raw mode, alternate screen, клавиш, resize и мыши;
- `ratatui` для layout/rendering;
- `std::fs`, `std::path::PathBuf` для файлов;
- `serde + toml` для конфига;
- `rand` для случайных чисел;
- обычные Rust-тесты для логики.

Так платформа остается деталью библиотек, а не всего приложения.

## 11. Этапы реализации

### Этап 1. Каркас и зависимости

- добавить зависимости в `Cargo.toml`;
- создать модули `domain`, `config`, `protocol`, `tui`;
- заменить `Hello, world!` на запуск приложения с обработкой ошибок;
- добавить минимальные unit-тесты.

### Этап 2. Настройки

- описать `Settings`;
- реализовать значения по умолчанию;
- реализовать чтение/запись `arithmet.toml`;
- добавить валидацию настроек;
- протестировать дефолты и невалидные значения.

### Этап 3. Бизнес-логика

- перенести `Operation`;
- перенести генератор примеров из `session.c`;
- перенести расчет оценки;
- описать `SessionResult` и `AnswerRecord`;
- покрыть генератор и оценку тестами.

### Этап 4. Протоколы

- перенести транслитерацию имени из `protocol.c`;
- реализовать выбор имени файла `000..999`;
- реализовать запись текстового протокола;
- добавить чтение списка старых результатов из каталога;
- добавить fallback-просмотр plain text.

### Этап 5. Минимальный TUI

- включить alternate screen и raw mode;
- нарисовать главный экран;
- реализовать переключение операций и редактирование настроек;
- реализовать старт сеанса;
- реализовать ввод ответа, таймер и `Tab` после ответа.

### Этап 6. Экран результатов

- показать результат последнего сеанса;
- добавить прокрутку по фактической высоте терминала;
- добавить просмотр старых протоколов из `results_dir`;
- добавить `F1` как вход в результаты.

### Этап 7. Полировка интерфейса

- улучшить адаптивную раскладку;
- добавить цвета и аккуратные блоки;
- добавить обработку resize;
- добавить мышь, если это не усложняет код;
- проверить Windows Terminal и Linux terminal.

### Этап 8. Проверка совместимости

- сравнить с C-версией по горячим клавишам;
- сравнить формат сохраненного протокола;
- проверить старые файлы из `examples/sessions`;
- проверить русское имя и транслитерацию;
- проверить таймер и служебные ответы `Esc`, `F10`, timeout.

## 12. Рекомендуемый порядок для новичка в Rust

Не начинать с TUI. Самый надежный порядок:

1. `Settings` и конфиг.
2. `Operation`, `Exercise`, генерация примеров.
3. Оценка и `SessionResult`.
4. Протоколы.
5. Только потом TUI.

Причина простая: TUI добавляет события, время жизни состояния, отрисовку,
resize и ошибки терминала. Если бизнес-логика уже работает и покрыта тестами,
интерфейс будет проще отлаживать.

## 13. Минимальный критерий готовности

Первая полноценная Rust-версия готова, когда:

- `cargo test` проходит;
- программа запускается в терминале;
- настройки читаются и сохраняются в TOML;
- можно пройти сеанс с теми же операциями, что в C-версии;
- горячие клавиши не хуже C-версии;
- итоговая оценка и протокол сохраняются;
- можно открыть последний и старые результаты из каталога;
- интерфейс не ломается при изменении размера терминала.
